# FFCA top-to-bottom audit — 2026-05-20

Audit covers: FFCA core package, FFCA agent, case-study pipeline, paper + slides. Lens: correctness, generality, design coherence, paper alignment.

Findings are bucketed by severity. Each item has a file:line ref and a proposed fix. Items with a `★` are correctness-critical — if any of these are silently broken, downstream numbers (and paper claims) are at risk.

## CRITICAL (correctness, paper-impact, or contradicts user-visible claims)

### C1 ★ Paper abstract + contributions still cite the retracted "100% recall on the three degraded experiments" claim
- **Where**: `main.tex:62` (abstract) and `main.tex:92` (contributions list).
- **What**: Both still say "the agent's diagnostic verdict identifies all three measurably-degraded experiments at 94% specificity and 100% recall" / "(recall 100%, specificity 94%)".
- **Why it's wrong**: `main.tex:375` (§7.3) explicitly retracts this: *"the rule-level diagnostic correlation we initially reported no longer holds … The original 100%-recall claim was an artefact of the rule misreading the seed axis as a time axis."*
- **Also banned vocabulary** ("degraded") in the abstract — the user's framing rule after the reset is to describe events factually, not bucket experiments.
- **Proposed fix**: rewrite the abstract sentence and the corresponding bullet around the Variant C/D/E retraining experiments. E.g.:
  > "On a real-world deployment problem — sub-meter compound flood forecasting with twenty retrained models — direct retraining experiments confirm the agent's affirmative recommendations: removing the identified load-bearing backbone hurts test-year RMSE on 20/20 experiments (+0.30 to +1.82 cm), while the two affirmative 'safe to drop' clusters both retrain within ±0.02 cm of the original."
- **Effort**: ~10 lines of LaTeX. Push to Overleaf.

### C2 ★ `aggregation.ensemble_trust_decisions` will never label a feature CONFIDENTLY PRUNE — string mismatch
- **Where**: `FFCA_package/ffca/core/aggregation.py:167`.
- **What**: branch reads `if I <= imp_lo and A >= agreement_high and modal_name == "Noise Candidate":` but `modal_name` is drawn from `ARCHETYPE_NAMES` (`core/archetypes.py:33-42`), whose archetype-0 string is `"Noise"`, not `"Noise Candidate"` (the paper name lives in the agent at `ffca_agent/archetypes.py:11`).
- **Effect**: in ensemble mode, no feature ever gets `CONFIDENTLY PRUNE`. Downstream: `silent_features_present`, `trust_prune_recommended`, the §7 "safe to drop" narrative all read from `trust.confident_prune.count`. The Variant C numbers came out clean because Co-Sens (not trust score) drove them, but **trust-driven prune recommendations are silently empty in ensemble mode** — a latent bug masquerading as conservatism.
- **Proposed fix**:
  ```python
  if I <= imp_lo and A >= agreement_high and modal_name == "Noise":
  ```
  (Use the package name. Or — better — import `Archetype.NOISE` and compare ints to remove the string dependency entirely.)

### C3 ★ Agent `report.py` trust-bucket parsing drops ensemble's `INVESTIGATE (multi-modal seeds)` features
- **Where**: `FFCA_agent/ffca_agent/report.py:120-133`.
- **What**: the bucket-match loop checks `paper_dec in dec` where `paper_dec` is the literal string `"INVESTIGATE (unstable)"`. For ensemble-mode reports the decision string is `"INVESTIGATE (multi-modal seeds)"` — `"INVESTIGATE (unstable)"` is **not** a substring of `"INVESTIGATE (multi-modal seeds)"`, so the feature falls through and is never bucketed.
- **Effect**: in ensemble mode, `trust.investigate.count` is 0 even when features clearly disagree across seeds; `trust_multi_modal_seeds` (the new rule we added to fix the seed-vs-epoch bug) cannot fire because its trigger `trust.investigate.fraction > 0.5` looks at a count that the bucketer just silently zeroed. This is exactly the rule that is supposed to be the corrected diagnostic for seed-axis cases; it is dead-coded by the bucketer.
- **Proposed fix**: match on the prefix word only, not the parenthetical suffix:
  ```python
  bucket_keys = {
      "CONFIDENTLY KEEP":    "confident_keep",
      "KEEP":                "keep_stable",
      "MONITOR":             "monitor",
      "INVESTIGATE":         "investigate",
      "CONFIDENTLY PRUNE":   "confident_prune",
  }
  for fname, dec in feature_trust.items():
      for prefix, snake in bucket_keys.items():
          if dec.startswith(prefix):
              buckets[snake].features.append(fname)
              buckets[snake].count += 1
              break
  ```
  (Order matters here too — `CONFIDENTLY KEEP`/`CONFIDENTLY PRUNE` must be checked before bare `KEEP`/`PRUNE`. Python dicts preserve insertion order; keep `CONFIDENTLY KEEP` first.)
- **Then**: add a regression test in `tests/test_evaluator.py` that builds a report with `INVESTIGATE (multi-modal seeds)` and asserts `ctx.trust_buckets["investigate"].count > 0`.

### C4 ★ `run_ffca_pruned.py` does NOT pass `mode="ensemble"`
- **Where**: `compound_flooding/run_ffca_pruned.py:519-527`. `FFCAReport(...)` is invoked with no `mode=` argument, so the default `"trajectory"` is used — even though the inputs are 30 seeds, not 30 epoch checkpoints.
- **Contrast**: the case-study drivers `run_ffca_ensemble_phaseB.py:146`, `run_ffca_ensemble_phaseB_full.py:181`, `run_ffca_counterfactual.py:143` all correctly pass `mode="ensemble"`.
- **Effect**: any FFCA report produced via `run_ffca_pruned.py` after the bug fix is still trajectory-mode on a seed ensemble — the very bug §7.3 is built around. The §7 results survive because the paper's reports came from the ensemble-mode drivers in `FFCA_agent/case_studies/`, **not** from this script. But `run_ffca_pruned.py` lives in the canonical case-study directory; a future re-run will silently regress.
- **Proposed fix**: pass `mode="ensemble"` on `report.py:519`; add a guard that raises if the number of checkpoints is `N_ENSEMBLE` and `mode != "ensemble"`. Or delete `run_ffca_pruned.py` entirely if the ensemble drivers supersede it.

### C5 ★ Pruning protocol auto-prunes ensemble-mode INVESTIGATE features
- **Where**: `compound_flooding/run_ffca_pruned.py:48,278-297`.
- **What**: `KEEP_DECISIONS = {"CONFIDENTLY KEEP", "KEEP (stable)", "MONITOR (borderline)"}` — anything outside this set gets pruned. In ensemble mode the decision is `INVESTIGATE (multi-modal seeds)`, which is not in `KEEP_DECISIONS`, so it gets auto-pruned. But the entire point of the ensemble-mode rule is that *INVESTIGATE in seed-axis mode is NOT safe to prune* (memory: `project_rulebook_bug_seed_vs_epoch.md`).
- **Effect**: the aggressive-pruning baseline that §7.1 critiques is silently re-enacted any time this script is run on an ensemble-mode report.
- **Proposed fix**: explicitly enumerate the new label or use a "prefix not in {INVESTIGATE, CONFIDENTLY PRUNE}" predicate. Better: drive pruning from the agent's recommendation, not from a hard-coded `KEEP_DECISIONS` set in the case-study script.

## HIGH (bugs that affect interpretation but not headline numbers)

### H1 Archetype classifier order biases against `NONLINEAR_DRIVER`
- **Where**: `FFCA_package/ffca/core/archetypes.py:84-100`.
- **What**: rules are first-match. `WORKHORSE` (line 89) requires only `i_r>0.7 ∧ v_r<0.3 ∧ x_r<0.3` — *no constraint on N*. A feature with high I, low V, low X, **high N** matches WORKHORSE first and never reaches the `NONLINEAR_DRIVER` test on line 93. This contradicts the binary code at `archetypes.py:55` (`[1,0,0,0]` for WORKHORSE).
- **Effect**: counts in archetype tables systematically underweight nonlinear drivers, overweight workhorses. The paper's archetype examples are unaffected (engineered data hits the right archetypes by construction) but real-world reports — including compound flooding — see this bias.
- **Proposed fix**: tighten WORKHORSE to `i_r>0.7 ∧ v_r<0.3 ∧ n_r<0.3 ∧ x_r<0.3`. Same for STABLE_CONTRIBUTOR. Then re-run the negative-control corpus to confirm the specificity numbers in §6 don't shift.

### H2 `TrustScore` only awards `CONFIDENTLY KEEP` to 3 of 8 stable archetypes
- **Where**: `FFCA_package/ffca/improvements_pkg/trust_score.py:91-97`. `dominant in (2, 3, 6)` covers WORKHORSE, CATALYST, STABLE_CONTRIBUTOR. A stable NONLINEAR_DRIVER (4), HIDDEN_INTERACTOR (1), VOLATILE_SPECIALIST (5), or COMPLEX_DRIVER (7) gets only `KEEP (stable)` even when its impact is high and stability is perfect.
- **Effect**: backbone-protection narrations under-recognise legitimate load-bearing nonlinear / interactive features. The agent's §7 backbone-identification verdicts work because compound-flooding `gwl` lags are workhorses, but this is a happy accident; on a more nonlinear domain the backbone would be quietly demoted.
- **Proposed fix**: high-importance + stable → `CONFIDENTLY KEEP` regardless of archetype identity. Use the trust score's `importance` field (already computed at line 88) plus a percentile threshold rather than a hard-coded archetype whitelist.

### H3 `TrustScore` decision tree ignores importance — low-impact noise gets mislabeled INVESTIGATE
- **Where**: `FFCA_package/ffca/improvements_pkg/trust_score.py:91-101`. Decision tree branches purely on `stability` and `dominant archetype`. A near-zero-impact feature whose archetype flips between Noise (0) and Stable Contributor (6) due to floating-point noise gets `INVESTIGATE (unstable)` instead of `CONFIDENTLY PRUNE`.
- **Effect**: trust buckets show false-positive INVESTIGATE on real models with many near-zero features. The rulebook's `silent_features_present` finding is suppressed.
- **Proposed fix**: when `importance < impact_p10` (or a similar low-quantile cutoff), override the decision to `CONFIDENTLY PRUNE` regardless of stability.

### H4 Ensemble-mode CoSensitivity runs on the *mean* gradient matrix, not the per-seed correlation structure
- **Where**: `FFCA_package/ffca/report.py:202-206` averages `_per_seed_gradients` then passes the averaged gradients to `CoSensitivityGroups.compute`.
- **What**: averaging gradient matrices before correlating destroys the very signal you wanted to keep — two seeds with opposite gradients on a feature cancel out, looking like a constant. The right ensemble extension is to either (a) compute one correlation matrix per seed and average those, or (b) concatenate per-seed gradient matrices into a single tall matrix (treating seeds as additional rows).
- **Effect**: Co-Sens groups in ensemble mode under-detect tight clusters. The compound-flooding cluster-prune validation (Variant C) still passes because it found a high-NC cluster, but the absence of a finding on the other 18 experiments is partly an artifact.
- **Proposed fix**: switch to row-concatenation: `np.vstack(self._per_seed_gradients)` instead of `np.mean(...)`. Document the semantics change in `aggregation.py`'s module docstring.

### H5 Many `feature.*` and `model.*` signals assume epoch axis but are not gated
- **Where**: `FFCA_agent/ffca_agent/report.py:217,223,237,266,270`.
- **What**: `feature.impact_epoch0`, `feature.impact_ratio_epoch0`, `feature.impact_saturation`, `model.checkpoint_drift_l2_pct`, `model.interaction_to_impact_growth_ratio` — all treat checkpoint[0] as "the start of training" and checkpoint[-1] as "the trained model". On a seed-axis ensemble these are arbitrary (which seed file got loaded first).
- **Cross-effect**: `data_leakage_immediate_dominance` (rulebook `:464-486`) triggers on `feature.impact_saturation > 0.85` with NO `applies_when` gate — so on a seed-axis case, if seed 1 happens to have similar impact to the ensemble median, the rule fires spuriously on every high-impact feature. `hierarchical_learning_confirmed` (`:376-393`) is the same story for `interaction_to_impact_growth_ratio`.
- **Proposed fix**: gate these signals in `report.py` — return `MissingSignal` when `checkpoint_kind != 'epoch'`. AND add `applies_when: case.checkpoint_kind == 'epoch'` to `data_leakage_immediate_dominance`, `hierarchical_learning_confirmed`, `overfitting_volatility_spike`, and any other rule that reads them.

### H6 `_check_applies_when` silently fires on un-parseable expressions
- **Where**: `FFCA_agent/ffca_agent/evaluator.py:205-231`.
- **What**: only handles `case.X == 'Y'` / `case.X != 'Y'`. Anything else (compound conditions, `case.X in (...)`, typos) returns `True`. A typo in YAML like `applies_when: "case.checkpoint_kind = 'epoch'"` (missing `=`) silently passes and the rule fires unconditionally.
- **Proposed fix**: when parse fails, raise (loudly) at rulebook load. The validator at `rulebook/validate.py` should also reject un-parseable `applies_when`. Add support for `in` and `and`/`or` while you're there — `case.checkpoint_kind in ('seed','mixed')` is a natural pattern that doesn't fit the current grammar.

### H7 Rulebook `trust_multi_modal_seeds` cannot fire because of C3
- **Where**: rulebook `:532-549`. Trigger `trust.investigate.fraction > 0.5` is unreachable as long as bug C3 zeros the `investigate` bucket in ensemble mode.
- **Effect**: the corrected seed-axis rule (the one we built specifically to fix the original bug) never fires. The original §7 finding that "neither variant of `trust_instability_high` fires on any of the twenty experiments" (line 375) may be partially driven by C3 rather than by genuinely-converged ensembles. Worth re-running once C3 is fixed.
- **Proposed fix**: fix C3, then re-run all 20 ensemble-mode rule evaluations and recompute the multi-modal rate. The 7.3% vs 12.2% comparison in §7.3 may need updating.

## MEDIUM (correctness in edge cases, design/coherence issues)

### M1 SE formula in CauchyHVP looks like it ignores the per-sample average
- **Where**: `FFCA_package/ffca/improvements_pkg/cauchy_hvp.py:169`.
- **What**: `se = (math.pi/2) * row_l1_np / max(math.sqrt(B), 1.0)`. `row_l1_np` is already the mean across `n_used` samples. The standard error of that mean should shrink by `1/sqrt(n_used)`, which the formula omits.
- **Effect**: CI widths reported in `interaction_ci` are too wide by `sqrt(n_used)` — over-conservative. Not a falsification, but makes interactions look less reliably positive than they are.
- **Proposed fix**: `se = (math.pi/2) * row_l1_np / max(math.sqrt(B * n_used), 1.0)`. Add a quick unit test on a known-Cauchy synthetic case.

### M2 `derivatives.py` and `cauchy_hvp.py` catch `RuntimeError` too broadly
- **Where**: `core/derivatives.py:81`, `improvements_pkg/cauchy_hvp.py:95`.
- **What**: `try: torch.autograd.grad(...) except RuntimeError:` is meant to catch "output independent of x" but also swallows e.g. CUDA OOM, graph-disconnected-in-unexpected-way, double-backward-not-implemented. All of these produce silent zeros instead of useful errors.
- **Proposed fix**: narrow to the specific error message string (`"element 0 of tensors does not require grad"` etc.), or pre-check `x.requires_grad` and skip the try/except in healthy paths.

### M3 Ensemble aggregation uses mean instead of median across seeds
- **Where**: `FFCA_package/ffca/core/aggregation.py:60-63`.
- **What**: a single outlier seed (e.g., bad initialization) can pull the mean by a lot. The paper itself uses *median* across the 30-seed ensemble for predictions (`feature_perturbation_retraining.py:225`). Inconsistent: predictions are median-aggregated but FFCA signatures are mean-aggregated.
- **Proposed fix**: switch the four-axis aggregation to median for consistency. Update §7 to note that aggregate signatures and aggregate predictions both use the median. (Re-running 20 experiments to recompute could affect tables — quantify the change first on one experiment before committing.)

### M4 Decision-string label `INVESTIGATE (multi-modal seeds)` missing from markdown legend
- **Where**: `FFCA_package/ffca/report.py:409-415`. `decision_meaning` dict lists 5 labels but not the ensemble-mode variant; the multi-modal-seeds rows in the report Markdown render with `—` as the explanation.
- **Proposed fix**: add `"INVESTIGATE (multi-modal seeds)": "seeds disagree on archetype — accept ensemble"`.

### M5 Rulebook `silent_features_present` blocked by bug C2
- **Where**: rulebook `:359-372`. Triggers on `trust.confident_prune.fraction > 0.10`. In ensemble mode `confident_prune` is permanently zero (because of C2), so this descriptor never fires even when noise features clearly exist. Downstream: the §7 narrations cannot cite this rule.
- **Proposed fix**: fix C2 first; this becomes live automatically.

### M6 `diagnostics.detect_capacity` recommends "train longer" — same epoch-axis assumption
- **Where**: `FFCA_package/ffca/diagnostics.py:315`. The `noise_frac > 0.50` recommendation is `"Train longer, or prune the noise features..."` — the exact phrasing the seed-axis bug taught us to avoid.
- **Effect**: in ensemble mode this still gets rendered into the markdown report (the agent rules don't fire on it because diagnostics.py is a separate path).
- **Proposed fix**: branch the recommendation on a `mode` parameter (the function is called from `run_all` at `diagnostics.py:506-535`; `mode` is already plumbed there). On `mode='ensemble'`: "These features are noise across every seed — safe to drop."

### M7 `signatures` `Workhorse` has same binary code as `Stable Contributor`
- **Where**: `FFCA_package/ffca/core/archetypes.py:47-57`. Both rows are `[1, 0, 0, 0]`. The similarity matrix at `:104-120` patches in a hard `0.85` similarity to compensate, but the underlying degeneracy means archetype assignment between these two depends purely on the percentile rules (which differ — `i_r>0.7` vs `i_r>0.5`) — not on the binary structure.
- **Coherence concern**: documentation and downstream consumers describe Workhorse vs Stable Contributor as semantically distinct, but the underlying 4D fingerprint is identical. Either differentiate the binary code (e.g., Stable Contributor = `[1, 0, 0, 0]` but with milder impact percentile) or rename the pair to make the relationship explicit.
- **Proposed fix**: documentation only — note in the dataclass docstring that the two archetypes are differentiated by Impact percentile, not by binary axis profile.

### M8 `archetype_feature_list` is O(n²) on large feature spaces
- **Where**: `FFCA_agent/ffca_agent/report.py:319-321`. Uses `self.feature_names.index(n)` inside a sort key. For compound flooding's 95 features this is fine; for a 10K-feature tabular report it would be slow.
- **Proposed fix**: build `{name: i for i, name in enumerate(self.feature_names)}` once at the top of the function.

### M9 Duplicate registries between `run_ffca_pruned.py` and `feature_perturbation_retraining.py`
- **Where**: `compound_flooding/run_ffca_pruned.py:56-182` defines `EXPERIMENTS` (with `hp` per experiment); `/tmp/feature_perturbation_bundle/feature_perturbation_retraining.py:117-176` re-defines the same registry split into `EXPERIMENTS` (name→lead) and `_HP` (name→hyperparameters).
- **Risk**: if hyperparameters get tuned in one place, the other drifts silently.
- **Proposed fix**: extract a single JSON or YAML manifest both scripts read at startup. Or, given the scripts are nearing end-of-life as one-shot drivers, document the relationship in a README and accept the duplication.

### M10 No constants module for trust-decision strings
- **Where**: strings like `"CONFIDENTLY KEEP"`, `"CONFIDENTLY PRUNE"`, `"INVESTIGATE (unstable)"`, `"INVESTIGATE (multi-modal seeds)"`, `"MONITOR (borderline)"` are duplicated across `FFCA_package/ffca/core/aggregation.py`, `improvements_pkg/trust_score.py`, `report.py`, `diagnostics.py`, and `FFCA_agent/ffca_agent/report.py`.
- **Risk**: this exact pattern caused bug C2 (one site wrote "Noise Candidate", another wrote "Noise") and bug C3 (one site wrote "(unstable)", another wrote "(multi-modal seeds)").
- **Proposed fix**: define an enum or constants module in `FFCA_package/ffca/core/decisions.py` and import everywhere. Bonus: type-checking can then catch the bucketing bug at lint time.

## LOW (nits, hardening, generality concerns without immediate impact)

### L1 `Y_BUFFER = 0.20` is a magic constant, undocumented origin
- **Where**: `run_ffca_pruned.py:50`, `feature_perturbation_retraining.py:63`. Both have it. Not derived; no comment on why 0.20.
- **Fix**: add a one-line comment citing where this came from (sigmoid output range during training).

### L2 Rulebook `linear_baseline_will_fail` thresholds are domain-specific
- **Where**: rulebook `:723-741`. Requires `hidden_interactor_pct > 5` AND `nonlinear_driver_pct > 5`. For an 11-feature dataset this is `>1 feature each` — almost always true.
- **Fix**: use count thresholds (`>=2` of each) instead of percentage, OR scale the percentage by feature count.

### L3 Heuristic thresholds in `archetype_imbalance_noise_dominant` / `_extreme` are not derived
- **Where**: rulebook `:225-253`. Thresholds 50% / 70% / 90% are documented as "heuristic; observed in compound-flooding pre-pruning runs". Generality across domains untested.
- **Fix**: the sensitivity-sweep in §6 (paper) shows these are fragile. Cite the sweep result in the rule's `confidence_factors` so downstream narrations can hedge.

### L4 `_eval_rule_at` aborts the whole rule on any missing signal
- **Where**: `FFCA_agent/ffca_agent/evaluator.py:246-249`. A rule with two triggers can't be configured "fire if trigger 1 is true and trigger 2 is missing".
- **Fix**: support `optional: true` per-trigger in the YAML schema, with "missing → treat as not-fired" semantics.

### L5 `_safe_format` returns the literal `{key}` for unresolved templates
- **Where**: `FFCA_agent/ffca_agent/evaluator.py:166-179`. A typo'd template key shows up verbatim in narrations.
- **Fix**: keep that behavior (it's intentional for graceful degradation), but also emit a warning to a side-channel log so devs notice.

### L6 `detect_data_leakage` z=3 threshold rarely fires in practice
- **Where**: `diagnostics.py:217-253`. `z_threshold=3.0` on Impact with ≤95 features means top ~1% — extremely strict. Plus `z_nlin < -0.5 AND z_int < -0.5` filters out legitimate Workhorses.
- **Fix**: relax to z=2.0 or make threshold configurable. Add a unit test on the App C.6 leakage case to confirm sensitivity.

### L7 `co_sensitivity.py` `abort_recommended` is misnamed
- **Where**: `improvements_pkg/co_sensitivity.py:181`. Boolean is `True` when "no group is safe to prune" — but `abort` suggests "abort the clustering", not "decline to recommend pruning".
- **Fix**: rename to `prune_recommended: not abort` (inverted polarity) or document the polarity clearly.

### L8 `n_features > 200` hard cutoff in `detect_tabular_shortcut`
- **Where**: `diagnostics.py:173-174`. Hard cap to disable the rule on image-channel cases. Will silently disable for 250-feature tabular models too.
- **Fix**: switch to checking `feature_shape` (1D vs 3D) only, or document the cap as a tunable.

### L9 Slides "Real-world test" section: no stale claims found
- Verified — slides do not mention "94% specificity / 100% recall" or "degraded experiments". §7 in slides aligns with the corrected variants C/D/E. No fix needed.

### L10 No regression test for the feature-ordering bug
- **Where**: `compound_flooding/run_ffca_pruned.py:310` has a comment saying *"Canonical regression test: 24h_gate ensemble median RMSE on the test year should be ~3.37 cm"* — but there's no actual test, just a comment.
- **Fix**: add a one-line script `tests/test_canonical_regression.py` that runs the 24h_gate evaluation and asserts `abs(rmse - 3.37) < 0.05`. Run in CI.

### L11 No tests in agent cover `applies_when` parsing
- **Where**: `FFCA_agent/tests/test_evaluator.py` likely doesn't test `_check_applies_when` failure modes.
- **Fix**: add tests for malformed `applies_when`, missing `case_meta`, compound conditions (currently expected to silently allow firing).

## Recommended ordering

If you only have time for a handful:
1. **C1** — paper abstract/contributions retraction. User-visible; trivial LaTeX edit.
2. **C2 + C3 + M5 + H7** — these are one tight cluster. Fix the two string mismatches and the entire ensemble-mode trust pipeline starts behaving correctly; the §7 numbers may then shift slightly (re-run worth doing).
3. **C4 + C5** — delete or correct `run_ffca_pruned.py` to prevent regression.
4. **H1 + H2** — archetype/trust over-restriction. Improves generality; affects table counts but not Variant C/D/E numbers.
5. **H4** — Co-Sens ensemble averaging. Better signal in future deployments; should be re-validated by running on the compound-flooding ensemble after the fix.
6. **H5 + H6** — gate epoch-axis signals + harden `applies_when` parsing. Prevents the next variant of the seed-vs-epoch bug.

Items below H6 are quality-of-life. Don't refactor for the sake of refactoring.
